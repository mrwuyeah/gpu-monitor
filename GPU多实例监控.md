# GPU 多实例监控使用说明

## 概述

支持同时运行多个 GPU 监控实例（不同端口），并通过一个统一的控制台页面管理所有实例的访问入口。

## 启动多个实例

### 启动第一个实例（监控 A）

```bash
python app.py --port 5000
```

### 启动第二个实例（监控 B）

```bash
python app.py --port 5001
```

> 每个实例独立采集所在机器的 GPU 数据，互不干扰。端口号可任意指定，只要不冲突即可。

### 指定监听地址

```bash
# 默认 0.0.0.0（允许局域网其他设备访问）
python app.py --port 5000 --host 0.0.0.0

# 仅本机访问
python app.py --port 5000 --host 127.0.0.1
```

## 控制台使用

每个实例都自带了控制台页面。

### 登录

访问 `http://localhost:{port}/console/login`

默认账号/密码：`admin / admin`

### 添加监控实例

1. 登录后进入控制台首页
2. 填写实例信息：
   - **名称**：给该实例起个标识名，如 "GPU服务器A"
   - **Base URL**：该实例的访问地址，如 `http://10.99.40.67:5000`
   - **Metrics URL（可选）**：vLLM metrics 地址，如 `http://10.99.40.67:8000/metrics`
   - **备注（可选）**：补充说明
3. 点击「保存」

### 管理实例

- **实时**：在新标签页打开该实例的实时监控页
- **历史**：在新标签页打开该实例的历史查询页
- **编辑**：修改已保存的实例信息
- **删除**：移除实例记录

## 每张 GPU 的 vLLM 进程信息

监控页面会为每张 GPU 展示其上的 vLLM 进程详情：

| 字段 | 说明 |
|------|------|
| PID | 进程 ID |
| 显存 | 该进程占用的 GPU 显存（MB） |
| 运行中 | 当前正在处理的请求数 |
| 排队中 | 当前排队的请求数 |
| KV缓存 | KV cache 使用率 |
| 模型 | 该 vLLM 实例部署的模型名称 |
| 命令行 | 进程启动命令 |

> vLLM 指标（运行中/排队中/KV缓存）通过访问 vLLM 的 `/metrics` 接口获取。系统会自动从进程命令行中提取 API 端口，默认回退到 8000 端口。

## 典型部署场景

### 场景一：单机多服务

同一台机器上运行多个 vLLM 实例（不同端口），一个 GPU 监控实例即可展示所有进程。

### 场景二：多机集群

每台机器运行一个 GPU 监控实例，用一台中心机器的控制台统一管理所有机器的访问入口：

```
中心控制台 (port 5000)
  ├─ 监控实例A → http://10.99.40.67:5000
  ├─ 监控实例B → http://10.99.40.68:5000
  └─ 监控实例C → http://10.99.40.69:5000
```

## FAQ

**Q: 控制台登录不上？**

A: 确认输入的是 `admin / admin`。输入框有默认值，直接点登录即可。如果 session 过期，清除浏览器 cookie 重新登录。

**Q: 页面显示"未检测到 vLLM 进程"？**

A: 说明 nvidia-smi 当前没有识别到 vLLM 计算进程。可能原因：
- vLLM 尚未启动或已停止
- nvidia-smi 需要 persistence mode 才能查询 compute apps
- 进程权限不足，`psutil` 无法读取进程信息

**Q: vLLM 指标显示 `-`？**

A: 系统无法连接到 vLLM 的 `/metrics` 接口。可在控制台为该实例配置正确的 Metrics URL，或确认 vLLM 进程的 `--port` 参数是否正确。

**Q: 数据库在哪个文件？**

A: `gpu_usage.db`（SQLite 文件），所有实例共用同一个数据库文件。
